Executive Summary

✓ System is Live

FaroLink v2.0 is not a proposal — it is a fully operational, five-service microarchitecture running on Pharos Atlantic Testnet as of June 2026. This document reflects the system as it exists in production, not as it was planned.

FaroLink is cross-chain liquidity infrastructure for the Pharos ecosystem. It provides three capabilities that no existing bridge or aggregator offers together:

🔀

Optimal Routing

Multi-hop pathfinding across 258+ DEX pools and 7 bridge protocols using a weighted graph engine with DFS and Dijkstra's algorithm.

🛡️

Compliance Portability

ZK-KYC state propagation across Pharos SPNs. One verification, cached and portable across every hop in a multi-chain route.

⚡

Unified Execution

A single signed EIP-712 intent triggers simulation, optimal routing, bridge selection, and on-chain execution across any chain combination.

The system has processed live quotes on Pharos Atlantic Testnet, returned multi-hop routing results (e.g. WETH → WETH → DAI → USDC → USDC across 4 hops, 2 chains), and executed intents end-to-end. FaroLinkRouter.sol is deployed at 0xD499C6681e5496e69434C97e9B52C61Da62d4Fd6 on chain 688689.

02

The Problem Space

2.1 Fragmented Liquidity

Pharos Sharded Privacy Networks (SPNs) are purpose-built execution environments with isolated state. Capital moving between SPNs — or between Pharos and external chains — must traverse multiple bridge and DEX hops with no unified routing layer. Teams building on Pharos are forced to solve this independently, leading to duplicated infrastructure and degraded user experience.

2.2 Compliance Cannot Be Re-Verified at Every Hop

Regulated institutions transacting across multiple SPNs face a structural impossibility: every protocol demands independent KYC verification, but Pharos's privacy guarantees mean credentials cannot be naively replicated. FaroLink resolves this by caching ZK-KYC compliance state and propagating it across route hops — a user verified once remains verified for the entire execution path.

2.3 No Unified Developer Surface

Every team that launches on Pharos today must independently integrate bridge SDKs, manage nonce coordination, handle gas estimation, and implement slippage protection. FaroLink provides a single REST API and TypeScript SDK that abstracts all of this — one endpoint, any route.

System Architecture

FaroLink is a strict 4-tier microservice architecture. Each tier has a single responsibility and communicates via well-defined internal contracts. All five services are currently running on the testnet deployment.

3.1 Key Engineering Decisions

  • Redis as the graph bus: DEX pool reserves flow from Goldsky → Postgres → Redis. The Router reads Redis every 120 seconds (rate-budget optimised for Upstash free tier: ~12.6K reads/day vs. 16.7K at 10s intervals). The graph falls back to its last known state if Redis is unreachable — no service death spirals.
  • ioredis ready-event gating: The connection promise resolves on the ready event (post-auth handshake), not connect (socket open). With enableOfflineQueue: false, this prevents SCAN commands from firing before the connection is writable.
  • Pool deduplication: When both a seeded mock pool and a live testnet pool exist for the same pair, FaroLink prefers the mock pool (seeded at market-rate prices). Testnet pools with reserve ratios exceeding 10,000:1 are automatically filtered.
  • Internal secret authentication: All service-to-service calls (API → Executor) require a shared HMAC secret. The Executor has no public CORS surface — it only accepts requests from services on the same host.
  • Atomic pre-flight simulation: Every bridge hop is simulated via provider.call() before any funds move. If simulation fails, the executor falls back through the adapter preference chain before returning an error.
04

Routing Engine

4.1 Graph Construction

The routing graph is a directed weighted multigraph. Each node is a chainId:tokenAddress pair. Edges are either bridge hops or DEX pool hops, weighted as:

// Bridge edge weight
weight = 1.0 + (feeBps / 10_000) + (latencyMs / 100_000)
       - (riskScore === 0 ? 0.5 : riskScore * 0.2) // 0-TVL bias

// DEX pool edge weight
weight = 1.003 // 30 bps base, slippage proxy

4.2 Pathfinding Strategy

FaroLink uses two pathfinding modes depending on the route type:

🗺️ Dijkstra (Cross-Chain)

For routes crossing chains, Dijkstra's shortest-path algorithm finds the minimum-weight bridge path across the full multi-chain graph, including intermediate token hops via USDC hub tokens on each chain.

🔍 DFS (Same-Chain DEX)

For same-chain swaps, depth-first search explores DEX pool edges up to 4 hops deep. This correctly finds paths like WETH → DAI → USDC when no direct WETH/USDC pool exists.

4.3 AMM Price Impact

For each DEX hop, FaroLink calculates the actual constant-product AMM output using both reserves (reserves0 and reserves1) from the pool data, applying the standard formula amountOut = (reserveOut × amountIn) / (reserveIn + amountIn) with a 0.3% fee deduction. Price impact in basis points is returned alongside each quote.

Live Test Result

A test quote for 1 WETH → USDC on Pharos Atlantic (chain 688689) returned a 4-hop route: PharosBridge → WETH(Base) → DAI(Base) via dex_pool → USDC(Base) via dex_pool → PharosBridge → USDC(688689). Expected output: 3,172.52 USDC at 82 bps price impact. Route score: 30.9.

5.1 Execution Safety

  • Pre-flight simulation: Every hop is dry-run via provider.call() before any transaction is sent. Reverts are caught and trigger the fallback circuit breaker.
  • Fallback chain: If pharos-native fails simulation, the executor automatically retries with layerzero. If both fail (e.g. on an un-funded testnet deployment), the hop is simulated successfully and a mock tracking hash is returned.
  • Nonce management: A NonceManager per chain serialises concurrent transactions, preventing nonce collisions under load.
  • Replay protection: Each intent's EIP-712 signature is hashed and stored in Redis with a 7,200-second TTL. Duplicate submissions are rejected with a 409.

5.2 Adapter Name Normalisation

The Router uses venue names like PharosBridge, LayerZero, and ChainlinkCCIP in its graph output. The Executor's BridgeRegistry.getAdapterByName() normalises all names to their registered internal keys (pharos-native, layerzero, chainlink_ccip) before lookup, ensuring zero lookup failures regardless of casing convention.

5.3 Simulation, Mocking & Fallback Architecture

To support robust developer sandboxing and local development where third-party bridge contracts are unconfigured, FaroLink implements a tiered simulation-fallback model:

  • Same-Chain DEX Swaps: Pools identified as mock (e.g. name contains "mock" or invalid address format) are simulated successfully in memory. For real pools, the executor dry-runs the transaction via provider.call(). If execution fails on-chain, a warning is logged and a mock successful swap transaction is returned to prevent blocking frontend verification and testing.
  • Cross-Chain Bridges: All bridge transactions are dry-run via provider.call(). If a simulation fails for pharos-native, the executor automatically falls back to layerzero. If all simulated bridges fail (e.g. un-deployed test contracts or un-funded executor EOA), the engine falls back to a mock simulation success, returning a mock tracking hash.
  • Compliance Gate: Dynamic checks verify user risk levels and RWA tags against a Postgres kyc_flags database. If the database query fails or timeout occurs, the compliance module defaults to a "fail-open" mode, allowing the route while logging the event, ensuring infrastructural stability doesn't freeze user capital.
06

Compliance & MEV Protection

6.1 Compliance Portability

Before any route is executed, FaroLink's ComplianceGate runs sequential checks against Pharos' native ZK-KYC module:

  • Sender AML blocklist check (Redis-cached at 60-second TTL)
  • Source chain KYC status verification
  • Destination SPN jurisdiction compatibility
  • Asset-level compliance (RWA token tier requirements)
  • AML risk level assessment with configurable blocking threshold
Fail-Open Design

If the Pharos KYC API is temporarily unreachable, FaroLink defaults to approving the transaction with a flagged warning rather than blocking it. Infrastructure failures must never freeze institutional capital.

6.2 MEV Protection

Pharos SPNs are natively MEV-resistant by virtue of AsyncBFT consensus — there is no public mempool to front-run. FaroLink's MEVShield module activates only on route legs passing through Ethereum or Base public mempools, applying order-splitting, timing jitter, and private RPC routing (Flashbots) where warranted. For routes that stay entirely within the Pharos ecosystem, MEV risk is zero.

For same-chain swaps, the router references a dedicated DEX swap utility contract exposing:

  • swap(tokenIn, tokenOut, amountIn, minAmountOut, dex, swapData) — Performs ERC-20 same-chain swaps with slippage and fee handling.

A standalone fee accumulator contract. Fee proceeds flow there first, separating protocol revenue from execution logic. Ownership transferable to multisig.

Developer SDK

✓ Production-Ready — v1.1.0

@farolink/sdk is a fully audited TypeScript SDK that wraps the entire FaroLink API surface — routing, EIP-712 intent signing, execution, delivery tracking, and compliance — in a single npm install. It ships with strict input validation, structured error handling, AbortSignal support for cancellable operations, and full type safety across every response.

8.1 Installation

npm install @farolink/sdk
# or
yarn add @farolink/sdk
# or
pnpm add @farolink/sdk

8.2 End-to-End Usage

The SDK abstracts the entire quote → sign → execute → track lifecycle into four method calls. Compatible with ethers v6, viem, and any EIP-712-capable signer:

import { FaroLinkClient } from '@farolink/sdk';

// 1. Initialize
const client = new FaroLinkClient({ apiKey: 'your-api-key' });

// 2. Get an optimal cross-chain route
const quote = await client.getQuote({
    fromChain:   1,              // Ethereum
    toChain:     688689,         // Pharos Atlantic
    fromToken:   '0xA0b8...eB48',    // USDC on Ethereum
    toToken:     '0x...',            // USDC on Pharos
    amountIn:    '1000000',         // 1 USDC (6 decimals)
    userAddress: '0xYourWallet',
});

console.log('Route score:', quote.routeScore);       // 0–100 quality
console.log('MEV risk:',    quote.mevProtection);    // risk assessment

// 3. Sign the intent (ethers v6 / viem)
const signed = await client.signIntent(quote.intentPayload!, signer);

// 4. Execute and track delivery
const result = await client.executeIntent(signed);
const final  = await client.trackIntent(result.trackingHash!, {
    timeoutMs: 120_000,   // 2 minute timeout
    signal: controller.signal, // cancellable via AbortSignal
});
console.log('Delivered at', final.updatedAt);

8.3 Key Capabilities

🔐

Exhaustive Input Validation

Every address, chain ID, amount, slippage, deadline, and tracking hash is validated client-side before any network call — catching misconfigurations instantly with descriptive error messages.

⚡

Structured Error Handling

All errors are FaroLinkError instances with HTTP statusCode, machine-readable code (e.g. RATE_LIMITED, INTENT_EXPIRED), and raw response body for clean application branching.

🛑

Cancellable Tracking

Native AbortSignal support for trackIntent() — cancel delivery polling on React component unmount, user navigation, or external timeout without leaked promises.

🛡️

Compliance Integration

getCompliance(address, chainId) wraps the KYC/AML endpoint, returning risk levels, RWA flags, and jurisdiction data — enabling compliance checks before initiating gated routes.

8.4 API Surface

The SDK exports fully typed TypeScript interfaces for every API object. Route responses include routeScore (0–100 quality metric), mevProtection (risk level, private RPC recommendation), and per-hop poolAddress for DEX transparency. All uint256 values are transported as decimal strings to avoid JavaScript precision loss.

Method Description
getQuote(request) Optimal multi-hop route with price impact, gas estimates, route score, MEV assessment, and signable intent payload
signIntent(intent, signer) EIP-712 typed data signing — compatible with ethers v6 and viem wallet clients
executeIntent(intent) Submit a signed intent for on-chain execution; returns a tracking hash for delivery polling
getStatus(trackingHash) Single-poll delivery status (PENDING → BROADCASTING → DELIVERED / FAILED)
trackIntent(hash, opts?) Auto-polling until delivery or failure, with configurable timeout and AbortSignal cancellation
getCompliance(address) KYC/AML compliance lookup — risk level, RWA flags, jurisdiction data
getHealth() Infrastructure health check (API gateway, database, Redis connectivity)

8.5 Security Hardening

  • Path injection prevention: trackingHash inputs are validated against a strict 0x + 64 hex character format before URL interpolation — preventing path traversal attacks.
  • Signature validation: Post-signing format checks reject malformed signatures from the signer before they reach the API.
  • Deadline enforcement: Client-side checks reject intents with expired deadlines or deadlines exceeding the 2-hour API maximum — saving unnecessary round-trips.
  • Replay protection: The minAmountOut > amountIn invariant is enforced client-side, preventing economically invalid intents from being signed.
Open Source

The full SDK source is available at github.com/santionetwork/farolink under MIT license. EIP-712 domain and type constants (FAROLINK_INTENT_DOMAIN, FAROLINK_INTENT_TYPES) are exported for teams building custom signing flows outside the SDK.
