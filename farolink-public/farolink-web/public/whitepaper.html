Section 02

The Problem Space

2.1 Liquidity Fragmentation

Pharos SPNs are purpose-built execution environments with isolated state. Capital moving between SPNs — or between Pharos and external chains — must traverse multiple bridge and DEX hops with no unified routing layer. Teams building on Pharos are forced to solve this independently, leading to duplicated infrastructure and degraded user experience across the ecosystem.

2.2 Compliance Cannot Be Re-Verified at Every Hop

Regulated institutions transacting across multiple SPNs face a structural impossibility: every protocol demands independent KYC verification, but Pharos's privacy guarantees mean credentials cannot be naively replicated. Without a compliance portability layer, institutional capital either cannot flow into Pharos RWA markets or does so in violation of internal compliance policy.

2.3 No Unified Developer Surface

Every team launching on Pharos today must independently integrate bridge SDKs, manage nonce coordination, handle gas estimation, and implement slippage protection. The compounded integration cost is measured in weeks per team. FaroLink provides a single REST API and TypeScript SDK that abstracts all of this — one endpoint, any route.

System Architecture

FaroLink is a strict 4-tier microservice architecture. Each tier has a single responsibility and communicates via well-defined internal contracts. All five services are currently running on Pharos Atlantic Testnet.

ServiceRole & Status
L4 — API Gateway (:4000)REST entry point. EIP-712 verification, tiered rate limiting, Zod payload validation, Swagger docs. ✓ Live
L2 — Router Engine (:3000)Weighted directed graph. DFS (same-chain DEX) + Dijkstra (cross-chain). 258+ pools indexed. ✓ Live
L3 — Execution Cluster (:5000)Pre-flight simulation, 7 bridge adapters, nonce management, fallback circuit breaker. ✓ Live
L1 — Indexer (:3001)Goldsky Postgres polling, compliance monitoring, Redis cache writes. ✓ Live
InfrastructureUpstash Redis (edge cache) · Neon PostgreSQL (persistent state) · Pharos RPC

3.1 Key Engineering Decisions

  • Redis as the graph bus: DEX pool reserves flow from Goldsky → Postgres → Redis. The Router reads Redis every 120 seconds, rate-budget-optimised for Upstash's free tier (~12.6K reads/day vs 16.7K at 10s intervals). The graph falls back to its last known state if Redis is unreachable — no service death spirals.
  • ioredis ready-event gating: The connection promise resolves on the ready event (post-auth handshake), not connect (socket open). With enableOfflineQueue: false, this prevents SCAN commands from firing before the connection is writable.
  • Pool deduplication: When both a seeded mock pool and a live testnet pool exist for the same pair, FaroLink prefers the mock pool (seeded at market-rate prices). Testnet pools with reserve ratios exceeding 10,000:1 are automatically filtered.
  • Internal secret authentication: All service-to-service calls (API → Executor) require a shared HMAC secret. The Executor has no public CORS surface — it only accepts requests from services on the same host.
  • Atomic pre-flight simulation: Every bridge hop is dry-run via provider.call() before any funds move. Simulation failures trigger the fallback adapter chain before returning an error.
Section 04

Routing Engine

4.1 Graph Construction

The routing graph is a directed weighted multigraph. Each node is a chainId:tokenAddress pair. Edges are bridge hops or DEX pool hops, weighted as follows:

Edge TypeWeight Formula
Bridge hop1.0 + (feeBps / 10,000) + (latencyMs / 100,000) − (riskScore === 0 ? 0.5 : riskScore × 0.2)
DEX pool hop1.003 (30 bps base fee as slippage proxy, refined per-hop by AMM output calculation)

4.2 Pathfinding Strategy

ModeApplication
Dijkstra's (cross-chain)Routes crossing chains. Finds minimum-weight bridge path across the full multi-chain graph, including intermediate USDC hub token hops on each chain.
DFS (same-chain DEX)Same-chain swaps. Explores DEX pool edges up to 4 hops deep. Correctly finds indirect paths (e.g. WETH → DAI → USDC) when no direct pool exists.

4.3 AMM Price Impact Calculation

For each DEX hop, FaroLink calculates actual constant-product AMM output using live pool reserves, applying the standard formula with 0.3% fee deduction. Price impact in basis points is returned alongside every quote. This is not an approximation — it uses the same mathematics as Uniswap V2's on-chain router.

Live Test Result — Pharos Atlantic Testnet

A quote for 1 WETH → USDC on Pharos Atlantic (chain 688689) returned a 4-hop route: PharosBridge → WETH(Base) → DAI(Base) via DEX pool → USDC(Base) via DEX pool → PharosBridge → USDC(688689). Note: the 82 bps price impact and 30.9 route score reflect testnet pool conditions with artificially shallow liquidity — not production market depth. On mainnet pools with real liquidity, impact on a 1 WETH trade will be sub-5 bps.

Section 05

Bridge Abstraction Layer

FaroLink's Executor wraps seven distinct bridge protocols under a single BridgeAdapter interface. Each adapter implements prepareTx(), estimateFee(), and waitForDelivery(). Callers never interact with a specific bridge — FaroLink selects the optimal one per hop.

Bridge ProtocolNotes
PharosBridge (Hyperlane)Default for intra-Pharos routes. Lowest latency within ecosystem.
LayerZero v2Primary fallback for Pharos ↔ Ethereum and Pharos ↔ Base routes.
deBridge DLNNative liquidity network. Zero-slippage model for large stablecoin transfers.
Chainlink CCIPEnterprise-grade messaging. Used for regulated cross-chain asset transfers.
Circle CCTPUSDC burn-and-mint. Zero slippage on USDC transfers of any size.
Axelar GMPGeneral message passing. Fallback for chains not covered by the above.
WormholeFinal fallback. Widest chain coverage.

Supported chains: Pharos Atlantic (688689), Ethereum (1), Base (8453), Arbitrum (42161), Optimism (10), Polygon (137), BNB Chain (56).

5.1 Execution Safety Model

  • Pre-flight simulation: Every hop is dry-run via provider.call() before any transaction is sent. Reverts are caught and trigger the fallback circuit breaker.
  • Fallback chain: If pharos-native fails simulation, the executor retries with layerzero. If all configured bridges fail, the hop returns a tracked mock hash — ensuring developer sandboxes are never blocked by unfunded test contracts.
  • Nonce management: A NonceManager per chain serialises concurrent transactions, preventing nonce collisions under concurrent load.
  • Replay protection: Each intent's EIP-712 signature is hashed and stored in Redis with a 7,200-second TTL. Duplicate submissions are rejected with HTTP 409.
  • Adapter name normalisation: BridgeRegistry.getAdapterByName() normalises all venue name conventions to registered internal keys before lookup — zero lookup failures regardless of casing.
Section 06

Compliance & MEV Protection

6.1 Compliance Portability

Before any route is executed, FaroLink's ComplianceGate runs sequential checks. Compliance state is cached in Redis and propagated across route hops — an institution verified once remains verified for the entire execution path.

CheckImplementation
AML blocklistRedis SET lookup. O(1). 300-second TTL. Instant rejection for sanctioned addresses.
Source KYC statusPharos ZK-KYC API query with 300-second Redis cache per address per chain.
Destination SPN jurisdictionRules engine checks sender's jurisdiction against destination SPN's permitted list.
Asset-level complianceRWA token tier check: kycTier ≥ 2 required for RWA-flagged assets.
AML risk levelConfigurable threshold. "blocked" = hard reject. "high" = approve with audit log.
Fail-Open Design

If the Pharos KYC API is temporarily unreachable, FaroLink defaults to approving the transaction with a flagged warning rather than blocking it. Infrastructure failures must never freeze institutional capital. This reflects how regulated financial systems actually operate — a compliance service outage is not grounds to block a client's trade.

6.2 MEV Protection

Pharos SPNs are natively MEV-resistant by virtue of AsyncBFT consensus — no public mempool, no front-running possible. FaroLink's MEVShield module activates only on route legs passing through Ethereum or Base public mempools, applying order-splitting, timing jitter (0–3,000ms randomised), and private RPC routing via Flashbots. For routes that stay entirely within the Pharos ecosystem, MEV risk is architecturally zero.

8.2 Security Hardening

  • Path injection prevention: trackingHash inputs validated against strict 0x + 64 hex format before URL interpolation.
  • Signature validation: Post-signing format checks reject malformed signatures before they reach the API.
  • Deadline enforcement: Client-side checks reject intents with expired deadlines or deadlines exceeding the 2-hour maximum.
  • Replay protection: minAmountOut > amountIn invariant enforced client-side, preventing economically invalid intents from being signed.
  • Structured errors: All errors are FaroLinkError instances with HTTP statusCode, machine-readable code (e.g. RATE_LIMITED, INTENT_EXPIRED), and raw response body.
